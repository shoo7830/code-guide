기본 규칙

  • 가능한 규칙을 지켜 코드를 작성합니다. 만약 수정해야 할 내용이 있다면 팀내에서 논의하여 문서가 업데이트 될 수 있도록 합니다.
  • 얼마나 많은 작업자가 참여했는지와 무관하게 한명이 쓴 것처럼 보이는 것이 좋습니다.

HTML

HTML 문법

  • 공백 문자 네 개를 사용하는 하드탭을 사용합니다.
  • 속성(attr)값을 지정할 때에는 작은 따옴표 대신 항상 큰 따옴표를 사용합니다.
  • 단일 태그에서는 슬래시(/)를 사용하지 않습니다. (e.g. <br/> or <img/>)
  • 텍스트의 단락은 <p>태그를 사용합니다. 여러 개의 <br>태그를 사용하지 않습니다.
    • <!DOCTYPE html>
<html>
	<head>
		<title>Page title</title>
	</head>
	<body>
		<img src="images/logo.png" alt="Lezhin">
		<h1 class="hello-world">Hello, world!</h1>
	</body>
</html>

    HTML5 doctype

    모든 HTML 페이지의 시작에 이 문서 타입을 선언함으로써, 표준 모드를 사용하게끔 강제합니다.

    		
			<!DOCTYPE html>
			<html>
			<head>
			</head>
			</html>

    언어(lang) 속성

    문서의 루트 요소인 html에 lang 속성을 추가합니다.

  • 한국어: ko
  • 영어: en
  • 중국어: zh
    • 		
			<html 
			lang=
			"ko"
			>
			<!-- ... -->
			<body 
			class=
			"ko-KR"
			>
			<!-- ... -->
			</html>

    인코딩 설정

    문자열 인코딩을 명시적으로 선언함으로써, 콘텐츠를 빠르고 쉽게 정확한 방법으로 렌더링 되도록 합니다.

    		
			<head>
			<meta 
			charset=
			"UTF-8"
			>
			</head>

    IE 호환모드 설정

    인터넷 익스플로러는 <meta> 태그를 통해 어떤 버전의 인터넷 익스플로러를 사용해야 하는지를 명시함으로써 문서의 호환성을 지원합니다. edge mode를 사용하여 IE가 항상 최신 버전의 모드를 사용하여 문서를 렌더링하도록 지정합니다.

    <meta http-equiv="X-UA-Compatible" content="IE=Edge">

    CSS, JavaScript 삽입

    HTML5 명세에서는 CSS와 Javascript를 불러올 때 type이나 text/css 속성을 생략하도록 권고하고 있습니다. 이를 따라 간결하게 코드를 작성합니다.

    		
			<!-- External CSS -->
			<link 
			rel=
			"stylesheet" 
			href=
			"code-guide.css"
			>
			<!-- In-document CSS -->
			<style>
			/* ... */
			</style>
			<!-- JavaScript -->
			<script src="code-guide.js"></script>

    완벽함보다는 실용성을 추구

    HTML 표준을 준수하고 시맨틱한 문서를 작성하기 위해 노력하기는 하지만, 추가적인 노력이 필요하지 않은 범위내에서만 합니다. 최대한 간결한 코드를 사용하도록 합니다.

    속성(attr) 순서

    HTML 태그 속성들은 코드의 가독성 향상을 위해 가능한 아래의 순서를 지켜 작성합니다.

    • id
    • class
    • name
    • data-*
    • src, for, type, href, value
    • title, alt
    • aria-*, role

    id를 가장 먼저 오게 합니다. class는 콤포넌트의 재사용성을 높여주기 때문에 두번째로 오게 합니다.

    <a class="..." id="..." data-toggle="modal" href="#">
					Example link
				</a>

				<input class="form-control" type="text">

				<img src="..." alt="...">

    Boolean 속성

    불리언 태그 속성은 굳이 속성값을 지정할 필요가 없으므로 지정하지 않습니다. (XHTML에서는 속성값 지정이 필요했지만 HTML5에서는 그런 제약이 없어졌습니다.)

    <input type="text" disabled>

				<input type="checkbox" value="1" checked>

				<select>
					<option value="1" selected>1</option>
				</select>

    마크업 간결화

    모듈화를 고려하여 가능한 마크업은 간결하게 작성합니다.

    <!-- Not so great -->
				<span class="avatar">
					<img src="...">
				</span>

				<!-- Better -->
				<img class="avatar" src="...">

    스크립트 내의 마크업

    자바스크립트 안에 마크업을 쓰는 것인 콘텐츠를 찾고 수정하기가 힘들며 효율적이지도 않습니다. 따라서 가급적이면 피하도록 합니다.

    HTML5 아웃라인

    문서의 영역별로 적절한 헤딩태그를 추가하여 HTML5 아웃라인이 올바르게 구성되도록 합니다. HTML5 아웃라인에 대한 설명은 다음을 참조합니다.

    CSS

    CSS 문법

    • 공백 문자 네 개를 사용하는 소프트탭을 사용합니다.
    • 쉼표를 사용하여 선택자를 그룹핑하는 경우, 개행하여 한 줄에 하나씩 작성합니다.
    • 가독성을 높이기 위해 여는 중괄호 앞에 여백을 한칸 놓습니다.
    • 닫는 중괄호는 새로운 줄에 놓습니다.
    • 각 선언에서 : 뒤에는 공백 문자 하나를 포함시킵니다.
    • 각 선언은 한 줄에 하나씩 하도록 합니다. 그렇게 함으로써 좀 더 정확하게 에러를 확인 할 수 있습니다.
    • 모든 선언은 마지막에 세미콜론(;)으로 끝냅니다. 맨 마지막 선언에서는 생략가능하지만, 이후 발생할 수 있는 실수를 방지하기 위함입니다.
    • 콤마로 나뉜 다중 속성값들은 각각의 콤마 뒤에 공백 문자를 하나 포함시키거나 줄 맞춰 개행합니다. (e.g., box-shadow).
    • rgb(), rgba(), hsl(), hsla(), rect() 값들 안에서는 콤마로 나뉜다고 하더라도 공백 문자를 포함시키지 않습니다. 이 값들은 다중 속성값이 아니기 때문입니다.
    • 속성값이나 색상값들에 소숫점을 사용할 때 0.을 사용하지 않습니다. (e.g., 0.5대신 .5를, -0.5px대신 -.5px를 사용)
    • 축약이 가능한 16진수값의 경우 가능한 축약하여 사용합니다. (e.g., #ffffff 대신 #fff사용)
    • 가급적 속성값들에는 홑따옴표를 사용합니다. (e.g., input[type='text'])
    • 값이 0일 때는 가능한 단위를 생략합니다. (e.g., margin: 0px; 대신 margin: 0; 사용)
    /* Bad CSS */
				.selector, .selector-secondary, .selector[type=text] {
				padding:15px;
				margin:0px 0px 15px;
				background-color:rgba(0, 0, 0, 0.5);
				box-shadow:0px 1px 2px #CCC,inset 0 1px 0 #FFFFFF
				}

				/* Good CSS */
				.selector,
				.selector-secondary,
				.selector[type='text'] {
				padding: 15px;
				margin-bottom: 15px;
				background-color: rgba(0,0,0,.5);
				box-shadow: 0 1px 2px #ccc, inset 0 1px 0 #fff;
				}

    속성 선언 순서

    CSS 속성들은 성격에 따라 다음과 같이 분류해 볼 수 있습니다.

    1. 포지셔닝
    2. 박스 모델
    3. 타이포그라피
    4. 배경
    5. 테두리
    6. 기타

    포지션 관련 선언들을 가장 먼저 작성합니다. 요소의 역할과 특징을 가장 크게 나타내는 속성들이기 때문입니다. 다음으로 요소의 크기와 위치를 지정하는 박스모델 관련 속성을 작성합니다.

    이 외의 나머지 속성들은 앞의 두 분류에 영향을 주지 않고, 해당 요소의 내부에서의 변화만을 지정하기 때문에 나중으로 배치합니다.

    나머지 속성중 자주 사용되며 속성의 갯수가 많은 타입,배경,테두리 관련 속성들을 따라서 작성하고, 그 외 기타 속성들을 마지막에 작성합니다.

    .declaration-order {

				/* Positioning */
				position: absolute;
				top: 0;
				right: 0;
				bottom: 0;
				left: 0;
				z-index: 100;

				/* Box-model */
				display: block;
				float: right;
				width: 100px;
				height: 100px;

				/* Typography */
				font: normal 13px "Helvetica Neue", sans-serif;
				line-height: 1.5;
				color: #333;
				text-align: center;

				/* Background */
				background-color: #f5f5f5;

				/* Border */
				border: 1px solid #e5e5e5;
				border-radius: 3px;

				/* etc */
				opacity: 1;
				}

    미디어 쿼리 위치

    기본적으로 사용하는 desktop, phone 폴더 외에는 다른 미디어쿼리를 만들지 않습니다.

    그 외에 불가피하게 추가하는 미디어쿼리들은 가급적이면 관련 규칙들이 있는 자리에 모아 놓으십시오. 미디어쿼리를 한데 모아놓으려다보면 오히려 작성하고 찾기가 불편해집니다. 미디어쿼리 선언이 여러개가 되더라도 괜찮습니다.

    .element { ... }
				.element-avatar { ... }
				.element-selected { ... }

				@media (min-width: 480px) {
					.element { ...}
					.element-avatar { ... }
					.element-selected { ... }
				}

    단일 속성 작성

    어떤 규칙이 오직 하나의 선언만을 포함하는 경우에는 가독성과 빠른 편집을 위해 개행하지 않고 작성합니다.

    속성이 여러개가 될 때는 당연히 개행하여 작성합니다. 여러개의 속성을 사용 할 때는 개행하는 편이 보기에도 좋고, 에러 디버깅에도 좋습니다.

    /* Single declarations on one line */
				.span1 { width: 60px; }
				.span2 { width: 140px; }
				.span3 { width: 220px; }

				/* Multiple declarations, one per line */
				.sprite {
				display: inline-block;
				width: 16px;
				height: 15px;
				background-image: url(../img/sprite.png);
				}
				.icon           { background-position: 0 0; }
				.icon-home      { background-position: 0 -20px; }
				.icon-account   { background-position: 0 -40px; }

    축약형 표기

    축약형 표기를 주의 없이 사용할 경우 문제가 발생 할 수 있습니다. 다음 속성들에서 그런 일들이 자주 발생합니다.

    • padding
    • margin
    • font
    • background
    • border
    • border-radius

    축약형만을 사용하는 습관이 생기면, 축약형에서 필수로 지정해야 하는 모든 값들을 설정해야 하는 경우가 아닌 상황에서도 값을 지정하게 됩니다. 또한 이전에 사용했던 비축약 속성으로 지정했던 내용들을 덮어쓰는 경우가 발생할 수 있습니다. 때문에, 축약속성을 사용할 때는 주의하도록 합니다.

    /* Bad example */
				.element {
				margin: 0 0 10px;
				background: red;
				background: url("image.jpg");
				border-radius: 3px 3px 0 0;
				}

				/* Good example */
				.element {
				margin-bottom: 10px;
				background-color: red;
				background-image: url("image.jpg");
				border-top-left-radius: 3px;
				border-top-right-radius: 3px;
				}

    Sass 중첩 구문 사용

    무의미하게 중첩을 반복하는 것을 경계하십시오. 중첩이 많아지면 선택자의 우선순위 계산 문제를 만들기 쉽기 때문입니다. 선택자를 반복해서 작성하는 것을 피하는 용도로만 중첩을 사용하십시오.

    // Without nesting
				.table > thead > tr > th { … }
				.table > thead > tr > td { … }

				// With nesting
				.table > thead > tr {
				> th { … }
				> td { … }
				}

    Sass 계산식 사용

    가독성 확보를 위해 계산식에 괄호를 적절히 사용해 주십시오.

    // Bad example
				.element {
				margin: 10px 0 @variable*2 10px;
				}

				// Good example
				.element {
					margin: 10px 0 (@variable * 2) 10px;
				}

    주석

    주석이 너무 장황한 것은 좋지 않습니다. 보는 이의 이해를 도울 수 있을 만큼만 간결하게 작성하도록 합니다.

    /* Bad example */
				/* Modal header */
				.modal-header {
					...
				}

				/* Good example */
				/* Wrapping element for .modal-title and .modal-close */
				.modal-header {
					...
				}

    클래스 작명

    • 클래스 이름들은 소문자와 대시(-)를 사용하세요.언더스코어(_)나 카멜케이스는 사용하지 않습니다.
    • 가급적 축약형을 사용하지 않도록 합니다. .button대신 .btn을 쓰는 것과 같이 자주 사용되어 쉽게 의미를 유추 할 수 있는 축약어를 사용하는 것은 괜찮으나 .sc와 같이 의미를 파악하기 어려운 축약어를 사용해서는 안됩니다.
    • 가능한 클래스명은 짧고 쉽고 간결하게 작성하십시오.
    • 시각적 표현보다는 요소의 의미와 구조, 목적을 생각한 작명이 좋습니다.
    • 관련있는 요소를 그룹짓기 위해 접두어를 사용하는 것은 좋습니다.
    • 요소의 변화하는 상태를 나타내는 추가 클래스는 .is-*의 형태로 작명하는 것이 좋습니다.
    /* Bad example */
				.t { ... }
				.red { ... }
				.header { ... }

				/* Good example */
				.tweet { ... }
				.important { ... }
				.tweet-header { ... }

    선택자

    • 가급적 요소명 보다 별도의 클래스를 지정하여 스타일을 작성하세요.
    • 선택자는 가능한 짧게 작성하고, 중첩이나 띄어쓰기를 통한 부모요소 지정이 3회 이상 반복되지 않도록 합니다.
    • 쉼표를 사용하여 여러개의 클래스에 같은 내용을 지정할 때는 반드시 쉼표 후 개행합니다. 개행없이 쉼표 사용할 경우 수정시에 실수가 발생하는 일이 많기 때문입니다.
    /* Bad example */
				span { ... }
				.page-container #stream .stream-item .tweet .tweet-header .username { ... }
				.avatar { ... }

				/* Good example */
				.avatar { ... }
				.tweet-header .username { ... }
				.tweet .avatar { ... }

    구성

    • 컴포넌트 별로 코드를 모아서 작성하는 것이 좋습니다.
    • 계층구조의 순서에 따라 작업하세요.
    • 여백을 사용하여 코드의 가독성을 확보하면 코드를 시각적으로 정돈하는데 도움이 됩니다. 축약하려고 애쓰지 마십시오.
    • 여러개의 CSS 파일을 나눌 때, 페이지보다는 컴포넌트 별로 나누십시오. 페이지는 컴포넌트들이 이동하면 언제든지 바뀔 수 있습니다.
    /*
				 * Component section heading
				 */

				.element { ... }


				/*
				 * Component section heading
				 *
				 * Sometimes you need to include optional context for the entire component. Do that up here if it's important enough.
				 */

				.element { ... }

				/* Contextual sub-component or modifer */
				.element-heading { ... }

    웹접근성 확보

    장애를 가진 사용자들도 불편 없이 사용 할 수 있도록 웹접근성 원칙들을 지켜 코드를 작성합니다.

    • 디자인상 노출되어 있지 않은 헤딩도 코드상에 작성(class="a11y" 사용)
    • 컨텐츠 이미지에 대체텍스트 지정
    • Tab 버튼 사용시 포커스 노출 및 이동순서가 올바르게 되도록 작성
    • 저시력 사용자를 위해 충분한 명도대비 확보

    에디터 설정

    다른 이들과 협업시 일관성이 확보되도록 미리 에디터의 설정을 맞춰놓는 것이 좋습니다.

    • 하드 탭 설정을 공백 문자 4개로 합니다.
    • 파일 저장시 마지막에 오는 불필요한 공백문자들을 제거하게 합니다.
    • 파일 저장시 UTF-8 인코딩으로 저장하도록 합니다.
    • 파일의 맨 마지막은 줄바꿈으로 끝나게끔 합니다.
    • 더 자세한 설정은 주요 에디터별 설정사항을 참고합니다.
